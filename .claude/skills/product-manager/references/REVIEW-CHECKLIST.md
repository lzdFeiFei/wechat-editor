# 产品文档评审检查清单

本文档提供系统化的产品文档评审检查清单,帮助全面评估PRD、MRD、BRD等产品文档的质量。评审时根据文档类型和复杂度,选择性使用相关检查项。

---

## 使用说明

### 评审优先级

**🔴 必须检查** - 核心问题,直接影响产品成败
**🟡 强烈建议** - 重要问题,影响产品质量
**🟢 建议检查** - 优化项,提升文档和产品品质

### 检查方法

1. **系统性检查**：按照清单逐项检查,不遗漏关键点
2. **重点关注**：根据产品类型和阶段,重点关注相关项
3. **记录问题**：发现问题时,记录具体位置和改进建议
4. **优先级排序**：将发现的问题按优先级分类

## 一、战略层面检查

### 1.1 产品定位 🔴

- [ ] **目标明确**：产品目标是否清晰、具体、可衡量?
  - 避免模糊表述如"提升用户体验"
  - 应有具体指标如"提升用户留存率10%"

- [ ] **价值主张**：产品为用户提供的核心价值是什么?
  - 解决什么问题?
  - 相比现有方案有何优势?
  - 为什么用户要选择我们?

- [ ] **战略对齐**：产品与公司战略是否一致?
  - 支持公司哪些战略目标?
  - 是否偏离公司主航道?

### 1.2 目标用户 🔴

- [ ] **用户画像**：目标用户是否清晰定义?
  - 人口属性（年龄、性别、地域、职业）
  - 行为特征（使用频率、场景、习惯）
  - 心理特征（需求、痛点、期望）

- [ ] **用户规模**：目标用户规模是否估算?
  - 潜在用户数量
  - 可触达用户数量
  - 预期转化用户数量

- [ ] **用户分层**：是否区分主要和次要用户?
  - 核心用户是谁?
  - 不同用户的需求差异?

### 1.3 商业价值 🟡

- [ ] **商业目标**：商业目标是否明确?
  - 营收目标
  - 用户增长目标
  - 市场份额目标

- [ ] **ROI分析**：投入产出比是否评估?
  - 预期投入（人力、时间、成本）
  - 预期收益（营收、用户、品牌）
  - 回本周期

- [ ] **盈利模式**：商业模式是否清晰?
  - 如何变现?
  - 收费模式是什么?
  - 可持续性如何?

## 二、需求层面检查

### 2.1 需求合理性 🔴

- [ ] **真实需求**：是否是真实的用户需求?
  - 有用户调研数据支撑?
  - 有数据分析发现?
  - 是否是伪需求?

- [ ] **需求优先级**：需求优先级是否合理?
  - P0（必须）、P1（重要）、P2（可选）分级
  - 优先级判断依据是什么?
  - 是否符合MVP原则?

- [ ] **需求边界**：需求范围是否清晰?
  - 做什么,不做什么明确界定
  - 本期、下期划分合理
  - 是否过度设计或设计不足?

### 2.2 需求完整性 🔴

- [ ] **场景覆盖**：业务场景是否全面?
  - 正常场景是否覆盖?
  - 异常场景是否考虑?
  - 边界场景是否遗漏?

- [ ] **流程完整**：业务流程是否完整?
  - 流程入口在哪里?
  - 流程节点是否齐全?
  - 流程出口在哪里?

- [ ] **角色覆盖**：相关角色是否都考虑到?
  - 用户角色
  - 运营角色
  - 管理角色

### 2.3 需求可行性 🔴

- [ ] **技术可行**：技术上能否实现?
  - 是否有技术难点?
  - 技术方案是否成熟?
  - 是否需要技术预研?

- [ ] **资源可行**：资源是否充足?
  - 人力资源是否到位?
  - 时间是否充裕?
  - 预算是否足够?

- [ ] **依赖条件**：外部依赖是否满足?
  - 依赖的系统或接口是否就绪?
  - 需要的数据是否可获得?
  - 第三方服务是否可用?

### 2.4 需求一致性 🟡

- [ ] **逻辑自洽**：需求之间是否矛盾?
  - 功能A和功能B是否冲突?
  - 规则X和规则Y是否一致?

- [ ] **系统兼容**：与现有系统是否兼容?
  - 是否影响现有功能?
  - 数据结构是否兼容?
  - 用户习惯是否冲突?

## 三、设计层面检查

### 3.1 用户体验 🟡

- [ ] **易用性**：功能是否易于使用?
  - 操作步骤是否简单?
  - 学习成本是否低?
  - 是否符合用户习惯?

- [ ] **流畅性**：交互流程是否流畅?
  - 流程是否顺畅无阻?
  - 是否有不必要的步骤?
  - 跳转路径是否合理?

- [ ] **一致性**：设计是否一致?
  - 交互方式是否一致?
  - 视觉风格是否统一?
  - 文案表达是否统一?

- [ ] **反馈性**：操作反馈是否及时?
  - 点击是否有反馈?
  - 加载是否有提示?
  - 结果是否有通知?

- [ ] **容错性**：是否考虑用户误操作?
  - 重要操作是否有二次确认?
  - 是否可以撤销?
  - 错误提示是否友好?

### 3.2 交互设计 🟡

- [ ] **交互规范**：交互是否符合平台规范?
  - iOS Human Interface Guidelines
  - Android Material Design
  - Web交互规范

- [ ] **手势操作**：手势设计是否合理?
  - 常用操作是否便捷?
  - 是否符合用户习惯?

- [ ] **导航设计**：导航是否清晰?
  - 用户知道在哪里?
  - 用户知道如何去其他地方?
  - 返回路径是否明确?

### 3.3 视觉设计 🟢

- [ ] **视觉层级**：信息层级是否清晰?
  - 主要信息是否突出?
  - 次要信息是否弱化?

- [ ] **色彩使用**：色彩是否合理?
  - 是否符合品牌色?
  - 是否有视觉冲击?
  - 是否考虑色盲用户?

- [ ] **字体规范**：文字是否规范?
  - 字号大小是否合适?
  - 字体是否统一?
  - 行间距是否合理?

### 3.4 文案设计 🟡

- [ ] **文案清晰**：文案是否易懂?
  - 避免专业术语?
  - 表达是否简洁?
  - 是否有歧义?

- [ ] **文案友好**：文案是否友好?
  - 语气是否亲切?
  - 是否引导用户?
  - 错误提示是否友好?

## 四、技术层面检查

### 4.1 技术方案 🔴

- [ ] **架构设计**：技术架构是否合理?
  - 是否符合系统整体架构?
  - 是否考虑扩展性?
  - 是否有技术债务?

- [ ] **接口设计**：接口设计是否规范?
  - RESTful规范
  - 接口版本管理
  - 接口文档完整

- [ ] **数据设计**：数据库设计是否合理?
  - 表结构是否规范?
  - 索引是否合理?
  - 是否考虑数据迁移?

### 4.2 性能要求 🟡

- [ ] **响应时间**：性能指标是否定义?
  - 页面加载时间要求
  - 接口响应时间要求
  - 是否有性能测试计划?

- [ ] **并发能力**：并发量是否评估?
  - 预期并发量
  - 峰值并发量
  - 压力测试计划

- [ ] **数据量**：数据规模是否预估?
  - 预期数据量
  - 数据增长速度
  - 存储容量规划

### 4.3 安全性 🔴

- [ ] **数据安全**：数据是否安全?
  - 敏感数据是否加密?
  - 数据传输是否加密?
  - 数据备份策略?

- [ ] **权限控制**：权限是否严格?
  - 用户认证机制
  - 权限验证规则
  - API接口鉴权

- [ ] **防护机制**：是否有安全防护?
  - 防刷机制
  - SQL注入防护
  - XSS攻击防护

### 4.4 可维护性 🟢

- [ ] **代码质量**：代码是否规范?
  - 编码规范
  - 注释完整
  - 可读性好

- [ ] **日志监控**：日志是否完善?
  - 操作日志
  - 错误日志
  - 性能日志

- [ ] **应急预案**：是否有应急措施?
  - 回滚方案
  - 降级策略
  - 故障预案

## 评审输出建议

### 评审结论

根据检查结果,给出评审结论:

- **通过**：无重大问题,可以进入下一阶段
- **有条件通过**：有问题但不影响推进,需要在XX前修复
- **不通过**：有重大问题,需要修改后重新评审

### 问题分级

将发现的问题按优先级分类:

**🔴 P0（必须解决）**
- 阻塞性问题
- 核心逻辑错误
- 重大遗漏

**🟡 P1（强烈建议）**
- 重要问题
- 体验问题
- 可优化项

**🟢 P2（建议优化）**
- 次要问题
- 细节完善
- 锦上添花

### 后续跟进

- 明确问题责任人
- 约定修复时间
- 计划复审时间

## 使用技巧

1. **根据实际情况灵活使用**：不是每个检查项都必须检查,根据文档类型和项目阶段选择
2. **重点关注高优先级项**：时间有限时,优先检查🔴标记的项目
3. **结合上下文判断**：某些检查项在特定场景下可能不适用
4. **持续更新清单**：根据团队实践,不断完善和更新检查清单
5. **工具辅助**：可以制作检查表格或清单工具,提高评审效率
