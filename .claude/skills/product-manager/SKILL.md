# Product Manager Skill

## 概述
这个Skill提供资深产品经理的专业能力,帮助评审产品文档和创作高质量的PRD。当用户需要评审产品需求文档、创作PRD、分析产品功能、设计用户体验,或进行产品规划时,使用此Skill来提供系统化、专业化的产品管理支持。

## 核心工作流程

产品经理的工作通常遵循以下流程:

1. **需求收集与分析** → 理解用户需求和业务目标
2. **竞品研究与市场分析** → 了解市场和竞争态势
3. **产品设计与规划** → 定义产品功能和优先级
4. **PRD文档创作** → 撰写清晰完整的需求文档
5. **需求评审与迭代** → 与团队评审并优化方案
6. **跟进开发与上线** → 确保产品按期交付

### 工作模式选择

根据用户请求,选择对应的工作模式:

**文档评审模式** → 用户上传文档或提供文档内容,请求评审和反馈
- 按照<文档评审流程>执行
- 参考 `references/REVIEW-CHECKLIST.md` 进行全面检查

**PRD创作模式** → 用户请求创作新的产品需求文档
- 按照<PRD创作流程>执行
- 参考 `references/PRD-TEMPLATE.md` 了解标准结构
- 参考 `references/PM-BEST-PRACTICES.md` 确保质量

**产品咨询模式** → 用户咨询产品设计、功能规划、用户体验等问题
- 提供专业的产品思维和建议
- 必要时参考最佳实践文档

## 文档评审流程

### 第一步:理解文档背景

在评审前,确保理解:
1. **文档类型**:PRD、MRD、BRD、技术方案、用户故事?
2. **产品阶段**:新产品、功能迭代、优化改进?
3. **目标受众**:开发团队、设计师、运营、高层?
4. **业务目标**:要解决什么问题?服务谁?期望达成什么?

如果信息不完整,主动询问用户以获取必要上下文。

### 第二步:结构化评审

按照优先级进行系统化评审:

#### 🔴 核心问题（必须解决）
1. **目标与价值**:产品目标是否清晰?用户价值是否明确?
2. **需求完整性**:关键需求是否遗漏?业务场景是否覆盖?
3. **逻辑一致性**:需求之间是否矛盾?与现有系统是否冲突?
4. **可行性**:技术上能否实现?资源是否充足?

#### 🟡 重要问题（强烈建议）
5. **用户体验**:交互流程是否顺畅?是否符合用户心智模型?
6. **数据指标**:如何衡量成功?关键指标是否定义?
7. **竞品分析**:是否了解竞品?有何差异化优势?
8. **边界场景**:异常情况如何处理?边界条件是否考虑?

#### 🟢 优化建议（建议改进）
9. **文档质量**:描述是否清晰?格式是否规范?
10. **细节完善**:交互细节、视觉规范、文案示例是否充分?
11. **扩展性**:是否考虑未来扩展?架构是否灵活?

### 第三步:提供建设性反馈

使用结构化格式输出评审结果:

```markdown
# 产品文档评审报告

## 📋 文档信息
- 文档类型:[PRD/MRD/BRD/其他]
- 产品名称:[产品名称]
- 文档版本:[版本号]
- 评审时间:[日期]

## 🎯 总体评价
[简要总结文档的整体质量、完整性和主要发现,2-3句话]

**总体评分**:⭐⭐⭐⭐⭐ （1-5星）

## 🔴 必须解决的问题
[列出核心问题,每个问题说明影响和改进建议]

### 问题1:[问题标题]
**影响**:[说明这个问题的严重性和影响范围]
**建议**:[提供具体的解决方案]

## 🟡 强烈建议改进
[列出重要问题和优化建议]

## 🟢 可选优化建议
[列出次要的改进建议]

## 👍 文档亮点
[认可文档中的优秀部分和好的实践]

## 📝 详细评审意见
[按章节或模块组织的详细评论]

### 章节1:[章节名称]
- 评论内容...

## ✅ 改进优先级建议
1. 第一优先级:[立即处理的问题]
2. 第二优先级:[尽快处理的问题]
3. 第三优先级:[可以后续优化的问题]

## 🚀 下一步行动
[建议的具体行动步骤]
```

## PRD创作流程

### 第一步:需求探索

与用户交互,明确以下关键信息:

1. **产品定位**
   - 产品名称和简介
   - 目标用户是谁?
   - 解决什么问题?
   - 核心价值是什么?

2. **业务目标**
   - 商业目标（营收、增长、留存等）
   - 用户目标（提升效率、改善体验等）
   - 战略目标（市场地位、品牌影响等）

3. **功能范围**
   - 核心功能有哪些?
   - 优先级如何?（MVP vs 完整版）
   - 技术约束或限制?

4. **上下文信息**
   - 现有产品情况
   - 竞品情况
   - 用户反馈
   - 市场数据

**提问策略**:不要一次性问太多问题,采用渐进式提问,根据用户回答深入探索。

### 第二步:结构化创作

根据PRD类型选择合适的文档结构:

**标准PRD** → 参考 `references/PRD-TEMPLATE.md` 的完整结构
**精简PRD** → 适用于小功能,包含:需求背景、核心功能、验收标准
**技术PRD** → 增加技术架构、接口设计、数据库设计等技术细节
**用户故事** → 采用 "作为[角色],我想要[功能],以便[价值]" 格式

### 第三步:内容撰写最佳实践

**清晰性原则**
- 使用简洁明了的语言,避免模糊表述
- 用具体场景和示例说明需求
- 关键概念要有明确定义

**完整性原则**
- 覆盖所有必要的业务场景
- 说明正常流程和异常流程
- 包含边界条件和特殊情况

**可执行性原则**
- 需求描述要具体可执行,不能过于抽象
- 提供具体的验收标准
- 包含必要的交互细节和视觉参考

**可衡量性原则**
- 定义清晰的成功指标
- 设定可量化的目标值
- 说明数据采集和分析方式

### 第四步:质量检查

创作完成后,按照 `references/REVIEW-CHECKLIST.md` 进行自查:
- 逻辑是否清晰连贯?
- 需求是否完整无遗漏?
- 是否有矛盾或冲突?
- 是否可以直接交付开发?

## 产品思维方法论

### 用户视角
始终从用户角度思考:
- 用户真正的痛点是什么?
- 这个功能能为用户带来什么价值?
- 用户会如何使用这个功能?
- 用户使用过程中可能遇到什么问题?

### 数据驱动
用数据支撑决策:
- 定义关键指标（KPI）
- 设定目标值和监控方式
- 考虑如何收集和分析数据
- 基于数据迭代优化

### 商业思维
平衡用户价值和商业价值:
- 这个功能如何支持商业目标?
- 投入产出比如何?
- 是否有变现可能?
- 对品牌和口碑的影响?

### 技术理解
理解技术可行性和成本:
- 技术实现难度如何?
- 需要多少开发资源?
- 是否有技术风险?
- 是否可以分期实现?

## 常见文档类型指南

### PRD（产品需求文档）
**目的**:详细描述产品功能和需求,指导开发实现
**受众**:开发、测试、设计、项目经理
**核心内容**:需求背景、功能列表、交互流程、验收标准
**详细结构**:参考 `references/PRD-TEMPLATE.md`

### MRD（市场需求文档）
**目的**:描述市场机会和产品定位,支持产品决策
**受众**:高层管理、市场团队、产品团队
**核心内容**:市场分析、竞品分析、目标用户、产品定位

### BRD（商业需求文档）
**目的**:阐述商业价值和投资回报,获取资源支持
**受众**:高层决策者、投资人
**核心内容**:商业目标、价值分析、成本收益、实施计划

### 用户故事
**目的**:敏捷开发中描述用户需求的简洁方式
**受众**:Scrum团队
**格式**:作为[角色],我想要[功能],以便[价值]
**补充**:验收标准、故事点估算

## 产品文档写作技巧

### 需求描述
✅ **好的描述**:
- "用户点击'提交'按钮后,系统验证表单完整性,如果必填项未填写,在对应字段下方显示红色错误提示"

❌ **差的描述**:
- "需要有表单验证功能"

### 用户场景
✅ **好的场景**:
- "小明在上班地铁上打开App,想查看今天的待办事项。他点击首页的'今日任务'卡片,进入任务列表页面,看到3个待办任务按优先级排列。他点击第一个任务查看详情..."

❌ **差的场景**:
- "用户可以查看任务列表"

### 验收标准
✅ **好的标准**:
- "点击登录按钮后,如果用户名或密码错误,1秒内显示错误提示'用户名或密码错误',错误提示3秒后自动消失"

❌ **差的标准**:
- "登录失败要有提示"

## 特殊场景处理

### 处理模糊需求
当需求不清晰时:
1. 不要臆测,主动向用户确认
2. 提供多个可能的理解供用户选择
3. 用具体例子验证理解是否正确
4. 在PRD中明确说明假设和前提

### 处理需求冲突
当发现需求矛盾时:
1. 明确指出冲突所在
2. 分析冲突的影响
3. 提供解决方案建议
4. 由用户或团队做最终决策

### 处理技术限制
当遇到技术约束时:
1. 说明技术难点和限制
2. 提供备选方案
3. 权衡不同方案的利弊
4. 建议分期实现或降级方案

## 资源文档

### PRD-TEMPLATE.md
完整的PRD文档模板,包含:
- 文档信息（版本、作者、更新记录）
- 需求概述（背景、目标、范围）
- 市场与用户分析
- 功能需求详细说明
- 非功能需求（性能、安全、兼容性）
- 数据埋点与指标
- 里程碑与排期
- 附录（名词解释、参考资料）

**使用时机**:创作正式的、完整的产品需求文档时,参考此模板确保内容完整规范。

### REVIEW-CHECKLIST.md
详细的产品文档评审检查清单,包含:
- 战略层面（产品定位、目标用户、商业价值）
- 需求层面（需求完整性、逻辑性、可行性）
- 设计层面（用户体验、交互流程、视觉规范）
- 技术层面（技术方案、性能要求、安全考虑）
- 运营层面（数据指标、运营策略、增长机制）
- 文档层面（清晰性、完整性、规范性）

**使用时机**:评审产品文档时,参考此清单确保评审全面系统,不遗漏关键检查项。

### PM-BEST-PRACTICES.md
产品经理最佳实践和方法论,包含:
- 需求挖掘与分析方法（5W2H、KANO模型、用户访谈）
- 优先级判断框架（RICE、MoSCoW、价值/成本矩阵）
- 产品设计原则（易用性、一致性、容错性）
- 协作沟通技巧（需求宣讲、跨团队协作）
- 产品思维模型（用户思维、数据思维、商业思维）
- 敏捷开发实践（用户故事拆分、Sprint规划）

**使用时机**:需要专业的产品方法论指导,或者解决复杂的产品问题时,参考此文档获取最佳实践。

## 应用场景

使用此Skill当用户:
- 请求评审产品文档:"帮我评审这个PRD"、"这个需求文档有什么问题"
- 请求创作PRD:"帮我写一个XX功能的PRD"、"需要一份产品需求文档"
- 询问产品设计:"这个功能应该怎么设计"、"用户体验如何优化"
- 寻求产品建议:"如何提升用户留存"、"功能优先级怎么排"
- 上传产品文档并寻求反馈或改进建议
- 提到产品管理相关术语:"PRD"、"用户故事"、"需求分析"、"产品规划"
- 需要竞品分析或市场研究
- 需要定义产品指标或成功标准
